# Writing a follow-up generator

A **follow-up generator** is a Rails generator that extends an
already-installed seams engine without re-templating the whole
engine. The first-party showcase is
[`seams:auth:add_oauth_provider`](../lib/generators/seams/auth/add_oauth_provider/add_oauth_provider_generator.rb);
this document is for hosts (and contributors to seams itself) writing
the second.

If you've never seen the marker mechanism before, skim
[`INSERTION_POINTS.md`](INSERTION_POINTS.md) first — that document
specifies the format, this one covers the workflow.

---

## When you want one

Reach for a follow-up generator when the change is **additive** and
**localised**:

- Register one more event in an engine's `register_events` initializer.
- Add one more route in front of (or after) an existing route block.
- Add one more entry to a registry-style hash, array, or class-name list.
- Ship a pre-wired adapter alongside the abstract base class the
  engine already exposes (`Auth::OAuth::LinkedIn` next to
  `Auth::OAuth::Google`, etc.).

Reach for `bin/seams resolve --eject` instead when the change is
**destructive** — you want to rewrite a controller, change a mailer's
copy, or replace the engine's default scope. The eject CLI marks the
file as host-owned so subsequent `bin/seams <engine>` runs leave it
alone; insertion points don't apply.

If you find yourself wanting to splice into a method body, a model, a
controller, or a migration, you've probably picked the wrong tool —
check INSERTION_POINTS.md's "What NOT to insert into" section.

## The 30-second template

```ruby
# lib/generators/seams/<engine>/<verb>_<noun>/<verb>_<noun>_generator.rb
require "rails/generators"
require "seams/generators/follow_up_generator"

module Seams
  module Generators
    module Auth
      class AddPasskeysGenerator < Seams::Generators::FollowUpGenerator
        engine_name "auth"

        source_root File.expand_path("templates", __dir__)

        def assert_engine_present
          assert_marker_exists!(
            file:   "lib/auth/engine.rb",
            marker: "auth.engine.events"
          )
        end

        def register_event
          splice(
            file:    "lib/auth/engine.rb",
            marker:  "auth.engine.events",
            content: <<~RUBY
              Seams::EventRegistry.register("identity.passkey_added.auth", emitted_by: "Auth")
            RUBY
          )
        end

        def report_summary
          say ""
          say "  Auth passkey support added.", :green
          say "  Next steps: ...", :yellow
        end
      end
    end
  end
end
```

Run it from the host:

```bash
bin/rails generate seams:auth:add_passkeys
```

That's the whole shape. The four primitives the base class supplies
do the heavy lifting:

| Primitive | Purpose |
| --- | --- |
| `engine_name "<engine>"` (class macro) | declares which engine this targets — used by `engine_path` and the recovery hint in `assert_marker_exists!`. |
| `engine_path(relative)` | resolves a path inside the host's `engines/<engine>/`. |
| `splice(file:, marker:, content:, before: false, indent: nil)` | wraps `Splicer.splice_after_marker` (or `splice_before_marker`) and prints a single-line `splice` log. |
| `assert_marker_exists!(file:, marker:)` | fails fast with a recovery hint when the engine wasn't generated, or was generated by a pre-Wave-10 seams. |

## Walking through `add_oauth_provider`

The shipped showcase
([`add_oauth_provider_generator.rb`](../lib/generators/seams/auth/add_oauth_provider/add_oauth_provider_generator.rb))
does six things. They're a useful checklist for any new follow-up
generator.

1. **Argument validation** — `normalise_name` strips, downcases, and
   underscore-squeezes the user's input, then raises
   `Seams::GeneratorError` with a copy-pasteable example if the result
   is empty or doesn't match the lowercase-underscored regex.
2. **Pre-flight check** — `assert_engine_present` calls
   `assert_marker_exists!` against the canonical configuration marker.
   If the marker isn't there, the host gets a single message telling
   them to re-run `bin/rails generate seams:auth`.
3. **Net-new files** — `create_adapter` and `create_adapter_spec` use
   Thor's `template` (via `engine_path(...)`) to write a new adapter
   class and its spec. Templates use ERB-bound helpers (`snake_name`,
   `camel_name`, `upper_name`) for the various capitalisation forms.
4. **Splice into existing files** — `splice_configuration_entry`
   inserts the new provider's hash entry inside the existing
   `@oauth_providers = {}` literal, at the
   `auth.configuration.oauth_providers` marker.
5. **Don't splice where you don't need to** — the doc-comment block
   at the top of the generator explicitly says routes and the
   `_oauth_buttons.html.erb` partial don't get touched, because the
   engine already iterates `Auth.configuration.oauth_providers` at
   runtime. Document the splices you skip; future readers (and
   yourself in six months) will thank you.
6. **End-of-run summary** — `report_summary` prints the files
   created, the splices made, and a numbered "Next steps" list. The
   host shouldn't need to grep the source to know what just happened.

## Idempotency: the gift, the trap

The Splicer is idempotent by design — re-splicing the same content
under the same marker is a no-op. That makes follow-up generators
safe to re-run during development without manual cleanup.

What it doesn't protect you from:

- **Net-new files written via `template`.** Thor's `template` skips
  existing files by default. If a host hand-edited the file your
  generator created, re-running won't clobber the edits — but it also
  won't pick up template improvements. Consider `force: true` only if
  you've documented that the file is generator-owned.
- **Argument changes.** `add_oauth_provider linkedin` and
  `add_oauth_provider linkedin_v2` produce different files and
  different splices. The Splicer can't dedupe across argument values;
  that's the host's responsibility.
- **Cross-marker invariants.** If your generator splices into two
  markers (a route AND a configuration entry), the Splicer is
  idempotent for each splice individually, but a partial run that
  succeeds on one and fails on the other leaves the host in a
  half-spliced state. Fail fast in `assert_engine_present` rather
  than partway through the work methods.

## Testing your generator

Mirror the shipped showcase's spec
(`spec/generators/seams/auth/add_oauth_provider_generator_spec.rb`).
The shape works:

- `Rails.root` set to a tmp directory; copy a fixture engine into it.
- Run the generator with `Rails::Generators.invoke(...)` or
  `described_class.new([...]).invoke_all`.
- Assert: the new files exist; the splice landed at the right
  position; running it again produces no diff (idempotency).

For machinery the generator depends on (the Splicer itself,
`FollowUpGenerator#assert_marker_exists!`), the unit tests at
`spec/seams/generators/splicer_spec.rb` and
`spec/seams/generators/follow_up_generator_spec.rb` already cover the
contract. You don't need to re-test those.

## Discoverability — telling hosts your generator exists

Three places mention a follow-up generator to a host:

1. **The post-install message** in
   [`install_generator.rb`](../lib/generators/seams/install/install_generator.rb)
   — listed under "Other useful commands" so a host running
   `seams:install` sees the canonical generators alongside the
   follow-up ones.
2. **The host's `bin/seams help`** — the help block in
   [`bin_seams.tt`](../lib/generators/seams/install/templates/bin_seams.tt)
   lists the follow-up generators alongside `resolve`.
3. **This document plus the catalogue.** Whenever you add a follow-up
   generator that targets a marker not yet exercised, mention it in
   the "Purpose" line of that marker's entry in
   [`INSERTION_POINTS_CATALOGUE.md`](INSERTION_POINTS_CATALOGUE.md).

If your follow-up generator lives in a host (not in the seams gem
itself), the pattern is the same — Rails picks up generators from
`lib/generators/` automatically.
